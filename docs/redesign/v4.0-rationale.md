**Lo que el SDK puede hacer (y el protocolo solo prometГӯa):**
- Rechazar instanciaciГіn de agentes sin permisos (determinГӯstico)
- Bloquear gates fГӯsicamente hasta recibir veredicto (no solo como instrucciГіn)
- Medir tokens consumidos en tiempo real (no estimado al cierre)
- Detectar race conditions en ejecuciГіn paralela (AsyncLockManager)
- Validar autenticidad de mensajes inter-agentes (HMAC)
- Calcular presupuesto de tokens ANTES de instanciar (proactivo)

**Lo que el SDK NO puede hacer (y el protocolo define):**
- Forzar que un LLM ejecute `pytest-cov` en lugar de estimar. Eso sigue siendo contrato de prompt.
- Reemplazar los checklists de gates вҖ” el SDK los ejecuta, pero el contenido es del protocolo.
- Modificar las Reglas Permanentes вҖ” son invariantes en ambas capas.

## 1. CAMBIOS EN AGENTES

### 1.1 LogisticsAgent вҖ” Nuevo (complemento proactivo)

**Problema que resuelve:** En v3.2 la detecciГіn de saturaciГіn es reactiva вҖ” ocurre cuando el agente ya estГЎ al 80% de contexto. LogisticsAgent actГәa ANTES de instanciar, estimando el presupuesto necesario para cada tarea del DAG.

**PosiciГіn en el flujo:** Se activa en FASE 2, en paralelo con el entorno de control. Reporta al Master Orchestrator antes de que los Domain Orchestrators creen worktrees.

**Rol:** AnГЎlisis de complejidad y planificaciГіn de recursos. No toma decisiones de gate. No veta planes. Produce un `TokenBudgetReport` que el Master usa para ajustar el nГәmero de expertos o fragmentar tareas antes de que empiecen.

**CondiciГіn de activaciГіn:** Siempre en objetivos Nivel 2. Opcional en Nivel 1.

**Lo que NO hace:**
- No reemplaza la clasificaciГіn Nivel 1 / Nivel 2 de `CLAUDE.md`. Esa clasificaciГіn es la puerta de entrada y permanece intacta.
- No usa "niveles dinГЎmicos" para reclasificar tareas. Sus estimaciones son presupuesto de recursos, no criterio de riesgo.

**Cap de estimaciГіn (defensa contra inyecciГіn de complejidad):**
Un objetivo malicioso o mal formulado podrГӯa inflar el `TokenBudgetReport` para forzar fragmentaciГіn innecesaria o consumir presupuesto arbitrariamente. Los caps son absolutos y derivan de la clasificaciГіn Nivel 1/2 вҖ” el LogisticsAgent no puede superarlos:

| Nivel de tarea | Cap de tokens por tarea |
|---|---|
| Nivel 1 | 8.000 tokens |
| Nivel 2 (tarea simple, вүӨ 3 archivos) | 40.000 tokens |
| Nivel 2 (tarea estГЎndar) | 100.000 tokens |
| Nivel 2 (tarea compleja, вүҘ 10 archivos) | 200.000 tokens |

Si la estimaciГіn supera el cap, el LogisticsAgent aГұade un `WARNING_ANOMALOUS_ESTIMATE` al informe. El Master lo presenta al usuario como parte del TokenBudgetReport antes de continuar.

```python
# sdk/piv_oac/agents/logistics.py

# Caps absolutos por clasificaciГіn de nivel (derivan de CLAUDE.md, no del SDK)
TOKEN_CAPS = {
    "nivel_1":          8_000,
    "nivel_2_simple":  40_000,    # вүӨ 3 archivos afectados
    "nivel_2_standard":100_000,
    "nivel_2_complex": 200_000,   # вүҘ 10 archivos afectados
}

@dataclass
class TokenBudgetReport:
    objective_id: str
    tasks: list[TaskBudget]
    total_estimated_tokens: int
    total_estimated_cost_usd: float
    fragmentation_recommended: list[str]  # IDs de tareas que deben fragmentarse
    warnings: list[str]                   # WARNING_ANOMALOUS_ESTIMATE si cap superado

@dataclass
class TaskBudget:
    task_id: str
    estimated_tokens: int
    cap_applied: int           # Cap que aplica a esta tarea
    capped: bool               # True si la estimaciГіn fue recortada al cap
    file_count: int
    dependency_count: int
    recommended_expert_count: int
    fragmentation_required: bool  # True si estimaciГіn supera 60% de ventana por experto

class LogisticsAgent:
    """
    EstimaciГіn proactiva de recursos por tarea del DAG.
    Modelo: claude-haiku-4-5 (tarea mecГЎnica de estimaciГіn).
    Presupuesto propio: 3000 tokens (no consume del pool del objetivo).
    """

    def analyze_dag(self, dag: list[Task], specs: SpecContext) -> TokenBudgetReport:
        budgets = []
        warnings = []
        for task in dag:
            budget = self._estimate_task(task, specs)
            if budget.capped:
                warnings.append(
                    f"WARNING_ANOMALOUS_ESTIMATE: tarea {task.id} estimГі "
                    f"{budget.estimated_tokens} tokens (cap aplicado: {budget.cap_applied}). "
                    "Verificar que el alcance de la tarea sea coherente con el objetivo."
                )
            budgets.append(budget)
        return self._build_report(budgets, warnings)

    def _estimate_task(self, task: Task, specs: SpecContext) -> TaskBudget:
        file_count = len(task.affected_files)
        dep_count = len(task.dependencies)

        # EstimaciГіn mecГЎnica вҖ” no LLM aquГӯ, solo heurГӯstica calibrada
        base_tokens = len(task.description.split()) * 4  # aprox tokens/palabra
        file_factor = 1 + (file_count * 0.15)
        dep_factor = 1 + (dep_count * 0.05)
        raw_estimate = int(base_tokens * file_factor * dep_factor)

        # VerificaciГіn de estimaciГіn por LLM solo si supera 10_000
        # para evitar LLM en tareas mecГЎnicas simples
        if raw_estimate > 10_000:
            raw_estimate = self._refine_with_llm(task, raw_estimate)

        # Aplicar cap segГәn nivel y complejidad de la tarea
        cap = self._resolve_cap(task, file_count)
        estimated = min(raw_estimate, cap)
        capped = raw_estimate > cap

        return TaskBudget(
            task_id=task.id,
            estimated_tokens=estimated,
            cap_applied=cap,
            capped=capped,
            file_count=file_count,
            dependency_count=dep_count,
            recommended_expert_count=self._recommend_experts(estimated, file_count),
            fragmentation_required=estimated > (CONTEXT_WINDOW * 0.60),
        )

    def _resolve_cap(self, task: Task, file_count: int) -> int:
        """Selecciona el cap segГәn la clasificaciГіn de la tarea."""
        if task.nivel == "nivel_1":
            return TOKEN_CAPS["nivel_1"]
        if file_count >= 10:
            return TOKEN_CAPS["nivel_2_complex"]
        if file_count <= 3:
            return TOKEN_CAPS["nivel_2_simple"]
        return TOKEN_CAPS["nivel_2_standard"]

    def _recommend_experts(self, tokens: int, files: int) -> int:
        if tokens <= 5_000 and files <= 3:
            return 1
        elif tokens <= 15_000 and files <= 8:
            return 2
        return 3  # mГЎximo вҖ” si supera, recomendar fragmentaciГіn de tarea
```

### 1.2 ExecutionAuditor вҖ” Nuevo (auditorГӯa integral out-of-band)

**Problema que resuelve:** Si la ejecuciГіn principal falla, satura o es vetada, el AuditAgent (que vive dentro del flujo) puede no poder completar su reporte de cierre. ExecutionAuditor tiene presupuesto propio y opera fuera del pool general, por lo que siempre genera un reporte, incluso en fallo.

**Diferencia clave con AuditAgent:**
- AuditAgent: superagente del entorno de control, revisa trazabilidad de RFs, emite veredictos en gates, escribe logs de veracidad. Rol protocolar.
- ExecutionAuditor: observador pasivo que registra el comportamiento del sistema (Вҝse ejecutaron los gates? Вҝhubo vetos? ВҝquГ© agentes tuvieron irregularidades?). No interviene en gates. No emite veredictos.

**Nomenclatura:** ExecutionAuditor, no "ControlRole" ni "sudo". El nombre refleja la funciГіn sin implicar permisos escalados.

```python
# sdk/piv_oac/agents/execution_auditor.py

class IrregularityType(Enum):
    GATE_SKIPPED = "gate_skipped"
    GATE_BYPASSED = "gate_bypassed"
    PROTOCOL_DEVIATION = "protocol_deviation"
    TOKEN_OVERRUN = "token_overrun"
    CONTEXT_SATURATION = "context_saturation"
    UNAUTHORIZED_INSTANTIATION = "unauthorized_instantiation"

class ExecutionAuditor:
    """
    AuditorГӯa integral out-of-band.

    CaracterГӯsticas:
    - Presupuesto propio: 5000 tokens (modelo claude-haiku-4-5)
    - Se ejecuta en paralelo desde FASE 2 hasta FASE 8
    - No emite veredictos de gate вҖ” solo registra irregularidades
    - Genera reporte final SIEMPRE, incluso si la ejecuciГіn principal falla
    - Su reporte es insumo del AuditAgent en FASE 8, no sustituto
    """

    INDEPENDENT_BUDGET_TOKENS = 5_000
    MODEL = "claude-haiku-4-5"

    def __init__(self):
        self._execution_log: list[ExecutionEvent] = []
        self._irregularities: list[Irregularity] = []
        self._gate_registry: dict[str, GateRecord] = {}

    def record_event(self, event: ExecutionEvent) -> None:
        """Registra cada evento de ejecuciГіn (gate, instanciaciГіn, veto)."""
        self._execution_log.append(event)
        self._detect_irregularity(event)

    def _detect_irregularity(self, event: ExecutionEvent) -> None:
        # Detectar gate saltado: si hay merge sin registro de gate previo
        if event.type == EventType.MERGE and not self._has_gate_record(event.source):
            self._irregularities.append(Irregularity(
                type=IrregularityType.GATE_SKIPPED,
                description=f"Merge de {event.source} sin registro de gate previo",
                severity="CRITICAL",
            ))

        # Detectar sobreconsumo de tokens
        if event.tokens_used > event.budget_allocated * 1.2:
            self._irregularities.append(Irregularity(
                type=IrregularityType.TOKEN_OVERRUN,
                description=f"Agente {event.agent_id} excediГі presupuesto en 20%+",
                severity="WARNING",
            ))

    def generate_final_report(self) -> ExecutionAuditReport:
        """
        Genera reporte de ejecuciГіn.
        Se llama al finalizar (Г©xito, fallo o veto).
        Nunca falla вҖ” si hay error interno, reporta el error sin propagar excepciГіn.
        """
        try:
            return ExecutionAuditReport(
                total_events=len(self._execution_log),
                total_irregularities=len(self._irregularities),
                critical_irregularities=[i for i in self._irregularities if i.severity == "CRITICAL"],
                gate_compliance=self._calculate_gate_compliance(),
                tokens_per_agent=self._aggregate_tokens(),
                summary=self._build_summary(),
            )
        except Exception as e:
            return ExecutionAuditReport(
                error=f"ExecutionAuditor fallГі internamente: {e}",
                partial_log=self._execution_log[:50],
            )
```

# sdk/piv_oac/core/factory.py

class AgentFactory:
    """
    Punto Гәnico de creaciГіn de agentes.
    Singleton вҖ” garantiza que toda instanciaciГіn pasa por validaciГіn de permisos.

    Responsabilidades:
    1. Validar que el solicitante tiene permisos para el tipo de agente pedido
    2. Resolver el proveedor LLM segГәn configuraciГіn
    3. Pasar contexto heredado de forma segura (whitelist estricta)
    4. Registrar la instanciaciГіn en ExecutionAuditor
    """

    _instance: Optional["AgentFactory"] = None

    @classmethod
    def get_instance(cls) -> "AgentFactory":
        if cls._instance is None:
            cls._instance = cls._initialize()
        return cls._instance

    def create_agent(
        self,
        agent_type: str,
        objective_id: str,
        requester: RequestContext,
        parent_context: Optional[SafeAgentContext] = None,
    ) -> AgentBase:
        # 1. Validar permisos вҖ” falla fuerte si no autorizado
        self._permission_store.validate(requester, agent_type)

        # 2. Validar contexto heredado si hay padre
        if parent_context:
            parent_context = self._inheritance_guard.project(parent_context)

        # 3. Resolver proveedor
        provider = self._provider_registry.resolve(agent_type)

        # 4. Instanciar
        agent = self._registry[agent_type](
            objective_id=objective_id,
            provider=provider,
            parent_context=parent_context,
        )

        # 5. Registrar en ExecutionAuditor
        self._execution_auditor.record_event(ExecutionEvent(
            type=EventType.AGENT_INSTANTIATED,
            agent_id=agent.id,
            agent_type=agent_type,
            requester=requester.id,
            objective_id=objective_id,
        ))

        return agent

## 4. HERENCIA CONTROLADA DE CONTEXTO

### 4.1 Problema y soluciГіn

v4 propone herencia de contexto entre agentes padre e hijo. Esto es valioso para transmitir `objective_id`, `task_scope` y permisos bГЎsicos sin repasar todo el estado al LLM. Sin embargo, la implementaciГіn original (cadenas recursivas, atributos sin whitelist explГӯcita) introduce riesgo.

**v4 corregido aplica:**
- Whitelist explГӯcita de atributos seguros a heredar (SAFE_INHERIT)
- Un Гәnico nivel de delegaciГіn de contexto (sin cadenas recursivas)
- Firma de snapshot + TTL para detectar contexto expirado o manipulado

### 4.2 InheritanceGuard

```python
# sdk/piv_oac/core/factory.py (mГіdulo de herencia)

class InheritanceGuard:
    """
    Controla quГ© puede heredar un agente hijo de su padre.
    Principio: whitelist, no blacklist.
    """

    # Solo estos atributos pueden pasar de padre a hijo
    SAFE_INHERIT: ClassVar[frozenset] = frozenset({
        "objective_id",
        "task_scope",
        "execution_mode",
        "compliance_scope",
        "parent_agent_id",
    })

    # Nada mГЎs. Si no estГЎ en SAFE_INHERIT, no se hereda.
    # Permisos, capabilities, credenciales, api_keys: nunca heredados.
    # Los permisos del hijo los asigna PermissionStore, no el padre.

    MAX_INHERITANCE_DEPTH = 1  # Solo padre вҶ’ hijo. Sin cadenas.

    def project(self, parent_context: AgentContext) -> SafeAgentContext:
        """
        Proyecta solo los atributos seguros del contexto padre.
        Firma criptogrГЎfica + TTL de 30 minutos.
        """
        safe = {k: v for k, v in vars(parent_context).items()
                if k in self.SAFE_INHERIT}

        snapshot = SafeAgentContext(
            data=safe,
            created_at=datetime.now(),
            ttl=timedelta(minutes=30),
            signature=self._sign(safe),
        )
        return snapshot

    def validate_snapshot(self, snapshot: SafeAgentContext) -> bool:
        """Valida firma y TTL antes de permitir al hijo usar el contexto."""
        if datetime.now() - snapshot.created_at > snapshot.ttl:
            raise InheritanceExpired("Contexto heredado expirado")
        if not hmac.compare_digest(self._sign(snapshot.data), snapshot.signature):
            raise InheritanceTampered("Firma de contexto heredado invГЎlida")
        return True
```

## 5. SISTEMA DE GOBERNANZA Y PERMISOS

### 5.1 Permisos sin bypass de gate

**CorrecciГіn crГӯtica vs. spec original:** El grupo `admin` tenГӯa permiso `gate:*:bypass`. Esto contradice directamente el principio Gate Bloqueante de las Reglas Permanentes. En v4 corregido: **ningГәn grupo ni usuario puede bypasear gates**. Los gates solo pueden ser aprobados por los agentes responsables o por confirmaciГіn humana explГӯcita (Gate 3).

```python
# sdk/piv_oac/core/permission_store.py

class PermissionStore:
    """
    Sistema de permisos individuales y grupales.

    Regla fundamental: gate:*:bypass NO EXISTE como permiso vГЎlido.
    El Гәnico mecanismo de bypass de gate es la confirmaciГіn humana
    explГӯcita en Gate 3, y es responsabilidad del protocolo, no del SDK.
    """

    FORBIDDEN_PERMISSIONS: ClassVar[frozenset] = frozenset({
        "gate:*:bypass",
        "gate:security:bypass",
        "gate:audit:bypass",
        "gate:coherence:bypass",
        "gate:standards:bypass",
        "protocol:skip",
    })

    def grant(self, subject: str, permission: str, ttl_minutes: int = 30) -> None:
        if permission in self.FORBIDDEN_PERMISSIONS:
            raise PermissionDenied(
                f"Permiso '{permission}' es invГЎlido en PIV/OAC. "
                "Los gates no se pueden bypasear programГЎticamente."
            )
        expires_at = datetime.now() + timedelta(minutes=ttl_minutes)
        self._store[subject][permission] = expires_at

    def validate(self, requester: RequestContext, agent_type: str) -> None:
        """Verifica que el solicitante puede instanciar el tipo de agente."""
        required = f"agent:{agent_type}:create"
        if not self._has_valid_permission(requester, required):
            raise PermissionDenied(f"{requester.id} no autorizado para {agent_type}")

    def _has_valid_permission(self, requester: RequestContext, permission: str) -> bool:
        # Verificar TTL вҖ” permisos vencidos se tratan como no concedidos
        for subject in [requester.user_id, *requester.groups]:
            perms = self._store.get(subject, {})
            if permission in perms and perms[permission] > datetime.now():
                return True
            if "agent:*:create" in perms and perms["agent:*:create"] > datetime.now():
                return True
        return False
```

## 6. GATE ENFORCER вҖ” GATES EN EL SDK

El GateEnforcer es la representaciГіn en cГіdigo de los gates del protocolo. No los reemplaza: los ejecuta. El checklist sigue siendo el definido en `contracts/gates.md`. El GateEnforcer garantiza que:
1. El gate espera a todos los agentes responsables (barrera de sincronizaciГіn)
2. Registra el veredicto en StateStore (persistente, no en memoria del agente)
3. Bloquea fГӯsicamente el flujo hasta recibir veredicto

```python
# sdk/piv_oac/core/gate_enforcer.py

class GateType(Enum):
    GATE_2_PLAN = "gate_2_plan"         # Antes de worktrees
    GATE_1_COHERENCE = "gate_1"         # Merge expertos вҶ’ rama tarea
    GATE_2B_CODE = "gate_2b"            # Merge tarea вҶ’ staging
    GATE_3_PREPRODUCTION = "gate_3"     # Staging вҶ’ main

class GateEnforcer:
    """
    Ejecuta el gate bloqueante y registra el resultado.

    Reglas:
    - Todos los agentes responsables deben completar antes de emitir veredicto
    - El veredicto se almacena en StateStore (no en memoria)
    - Un rechazo incrementa el contador en StateStore (para escalado en 2do rechazo)
    - Gate 3 nunca ejecuta acciГіn automГЎtica вҖ” solo presenta al usuario
    """

    def __init__(self, state_store: StateStore, execution_auditor: ExecutionAuditor):
        self._store = state_store
        self._auditor = execution_auditor
        self._lock = AsyncLockManager()

    async def enforce(
        self,
        gate_type: GateType,
        objective_id: str,
        task_id: str,
        agents: list[AgentBase],
    ) -> GateVerdict:
        async with self._lock.acquire(f"gate:{task_id}:{gate_type.value}"):
            # Lanzar todos los agentes del gate en paralelo real
            verdicts = await asyncio.gather(*[
                agent.review(gate_type, task_id)
                for agent in agents
            ])

            # Consolidar: todos deben aprobar
            if all(v.approved for v in verdicts):
                result = GateVerdict(approved=True, gate=gate_type)
            else:
                result = GateVerdict(
                    approved=False,
                    gate=gate_type,
                    rejections=[v for v in verdicts if not v.approved],
                )

            # Persistir resultado
            await self._store.save(
                key=f"gate:{objective_id}:{task_id}:{gate_type.value}",
                value=result.to_dict(),
            )

            # Registrar en ExecutionAuditor
            self._auditor.record_event(ExecutionEvent(
                type=EventType.GATE_COMPLETED,
                gate=gate_type,
                task_id=task_id,
                verdict=result,
            ))

            # Incrementar contador de rechazos si aplica
            if not result.approved:
                await self._increment_rejection_count(objective_id, task_id)

            return result

    async def _increment_rejection_count(self, objective_id: str, task_id: str) -> None:
        key = f"rejection_count:{objective_id}:{task_id}"
        current = await self._store.load(key) or 0
        await self._store.save(key, current + 1)
        if current + 1 >= 2:
            # 2do rechazo consecutivo вҶ’ escalar al Master segГәn protocolo
            await self._store.save(
                f"escalation_required:{objective_id}:{task_id}", True
            )
```

## 11. VALIDACIГ“N CRIPTOGRГҒFICA INTER-AGENTES

La expiraciГіn de un mensaje y la manipulaciГіn de su firma son dos eventos de naturaleza radicalmente distinta y requieren respuestas diferentes:

- **Firma invГЎlida** вҶ’ ataque o corrupciГіn intencional вҶ’ `MessageTampered` вҶ’ escalar como `SECURITY_VIOLATION` inmediato, sin reintento.
- **Mensaje expirado** вҶ’ tarea de larga duraciГіn o latencia de red вҶ’ `MessageExpired` вҶ’ reintento automГЎtico con re-firma (mГЎx. 3 intentos, backoff 2s). Solo escala si los 3 fallan.

Tratar ambos casos como `SecurityError` genГ©rico, como hacГӯa el diseГұo anterior, genera falsos positivos de seguridad en gates de tareas largas e interrumpe ejecuciГіn legГӯtima.

```python
# sdk/piv_oac/core/crypto_validator.py

class MessageTampered(SecurityError):
    """Firma criptogrГЎfica invГЎlida вҖ” posible ataque o corrupciГіn. No reintentar."""

class MessageExpired(RetryableError):
    """Mensaje TTL vencido вҖ” volver a firmar y reenviar. MГЎx. 3 intentos."""

class CryptoValidator:
    """
    Valida autenticidad de mensajes entre agentes.
    Implementa Zero-Trust real: cada agente verifica independientemente.

    Uso: antes de que un agente procese el output de otro.
    Cobertura: mensajes de gate, reportes de subagentes, veredictos.

    PolГӯtica de errores:
    - MessageTampered вҶ’ SECURITY_VIOLATION inmediato, sin reintento
    - MessageExpired  вҶ’ reintento con re-firma, mГЎx. 3 intentos, backoff 2s
    """
    TTL_SECONDS = 300       # 5 minutos
    MAX_RETRIES = 3
    RETRY_BACKOFF_S = 2

    def sign(self, message: dict, sender_id: str) -> SignedMessage:
        payload = json.dumps(message, sort_keys=True)
        timestamp = datetime.now().isoformat()
        data = f"{sender_id}:{timestamp}:{payload}"
        signature = hmac.new(
            self._secret.encode(), data.encode(), hashlib.sha256
        ).hexdigest()
        return SignedMessage(payload=message, sender_id=sender_id,
                             timestamp=timestamp, signature=signature)

    def validate(self, signed: SignedMessage, expected_sender: str) -> None:
        # 1. Verificar identidad del emisor вҖ” fallo duro, no reintentar
        if signed.sender_id != expected_sender:
            raise MessageTampered(
                f"Sender mismatch: esperado '{expected_sender}', recibido '{signed.sender_id}'"
            )

        # 2. Verificar TTL вҖ” fallo suave, reintentable
        age = (datetime.now() - datetime.fromisoformat(signed.timestamp)).total_seconds()
        if age > self.TTL_SECONDS:
            raise MessageExpired(
                f"Mensaje de {signed.sender_id} expirado ({age:.0f}s > {self.TTL_SECONDS}s). "
                "El emisor debe re-firmar y reenviar."
            )

        # 3. Verificar firma HMAC вҖ” fallo duro, no reintentar
        data = f"{signed.sender_id}:{signed.timestamp}:{json.dumps(signed.payload, sort_keys=True)}"
        expected_sig = hmac.new(
            self._secret.encode(), data.encode(), hashlib.sha256
        ).hexdigest()
        if not hmac.compare_digest(expected_sig, signed.signature):
            raise MessageTampered("Firma HMAC invГЎlida en mensaje inter-agente")

    async def validate_with_retry(
        self,
        signed: SignedMessage,
        expected_sender: str,
        resigner: Callable[[], SignedMessage],
    ) -> None:
        """
        Valida con reintento automГЎtico ante MessageExpired.
        resigner: callable que genera un nuevo SignedMessage con timestamp fresco.
        """
        for attempt in range(self.MAX_RETRIES):
            try:
                self.validate(signed, expected_sender)
                return  # ValidaciГіn exitosa
            except MessageExpired:
                if attempt == self.MAX_RETRIES - 1:
                    raise  # Agotados los reintentos вҶ’ escalar al Master
                await asyncio.sleep(self.RETRY_BACKOFF_S)
                signed = resigner()  # Re-firmar con timestamp fresco
            except MessageTampered:
                raise  # Nunca reintentar ante firma invГЎlida
```

## 11b. CODE SIGNING PARA SKILLS

**Problema que resuelve:** El `SkillRegistry` carga archivos desde `/skills/` en disco con lazy loading. Sin verificaciГіn de integridad, un archivo modificado entre su escritura autorizada (StandardsAgent post-gate) y su carga por un agente pasarГӯa desapercibido, violando la regla permanente **Skills Inmutables**.

**Mecanismo:** Cada vez que StandardsAgent escribe o actualiza un skill (Гәnico evento autorizado, post gate SecurityAgent + confirmaciГіn humana), el SDK calcula el `sha256` del contenido y lo registra en `skills/manifest.json`. Antes de cargar cualquier skill, `AtomLoader` verifica el hash. Si no coincide вҶ’ `BLOQUEADO_POR_HERRAMIENTA`, notificar usuario.

**`skills/manifest.json` вҖ” estructura:**

```json
{
  "version": "1.0",
  "generated_at": "<ISO8601>",
  "signed_by": "StandardsAgent",
  "objective_id": "<id del objetivo en que se generГі>",
  "skills": {
    "orchestration": {
      "path": "skills/orchestration.md",
      "sha256": "a3f8c...",
      "last_updated": "<ISO8601>",
      "gate_verified": true
    },
    "cost-control": {
      "path": "skills/cost-control.md",
      "sha256": "b7d2e...",
      "last_updated": "<ISO8601>",
      "gate_verified": true
    }
  }
}
```

```python
# sdk/piv_oac/core/memory/atom_loader.py

class SkillIntegrityError(SecurityError):
    """Hash del skill no coincide con el manifest. Posible modificaciГіn no autorizada."""

class AtomLoader:
    """
    Carga skills con verificaciГіn de integridad SHA-256.
    Si el manifest no existe o el hash no coincide вҶ’ BLOQUEADO_POR_HERRAMIENTA.
    """

    MANIFEST_PATH = "skills/manifest.json"

    def __init__(self):
        self._manifest = self._load_manifest()
        self._cache: dict[str, str] = {}

    def load(self, skill_name: str, force: bool = False) -> str:
        """
        Carga un skill con verificaciГіn de integridad.

        Fases:
        1. Buscar en cachГ© (si no force)
        2. Verificar que el skill estГЎ en el manifest
        3. Cargar contenido del disco
        4. Verificar sha256 contra manifest
        5. Cachear y retornar
        """
        if not force and skill_name in self._cache:
            return self._cache[skill_name]

        # Verificar que estГЎ en el manifest
        entry = self._manifest.get("skills", {}).get(skill_name)
        if not entry:
            raise SkillIntegrityError(
                f"Skill '{skill_name}' no estГЎ en skills/manifest.json. "
                "Solo StandardsAgent puede registrar skills tras gate SecurityAgent."
            )

        # Cargar del disco
        skill_path = Path(entry["path"])
        content = skill_path.read_text(encoding="utf-8")

        # Verificar integridad
        actual_hash = hashlib.sha256(content.encode()).hexdigest()
        if not hmac.compare_digest(actual_hash, entry["sha256"]):
            raise SkillIntegrityError(
                f"Integridad violada en skill '{skill_name}'. "
                f"Hash esperado: {entry['sha256'][:12]}... "
                f"Hash real: {actual_hash[:12]}... "
                "BLOQUEADO_POR_HERRAMIENTA вҖ” notificar al usuario antes de continuar."
            )

        self._cache[skill_name] = content
        return content

    def register_skill(self, skill_name: str, path: str, content: str) -> None:
        """
        Registra un skill nuevo o actualizado en el manifest.
        Solo puede ser llamado por StandardsAgent (validado por AgentFactory vГӯa permisos).
        """
        skill_hash = hashlib.sha256(content.encode()).hexdigest()
        self._manifest.setdefault("skills", {})[skill_name] = {
            "path": path,
            "sha256": skill_hash,
            "last_updated": datetime.now().isoformat(),
            "gate_verified": True,
        }
        self._persist_manifest()
        # Invalidar cachГ© del skill actualizado
        self._cache.pop(skill_name, None)

    def _load_manifest(self) -> dict:
        manifest_path = Path(self.MANIFEST_PATH)
        if not manifest_path.exists():
            # Manifest ausente вҶ’ sistema en estado inicial o corrupto
            # Retornar manifest vacГӯo y dejar que load() falle al buscar skills
            return {"skills": {}, "version": "1.0"}
        return json.loads(manifest_path.read_text())

    def _persist_manifest(self) -> None:
        Path(self.MANIFEST_PATH).write_text(json.dumps(self._manifest, indent=2))
```

**IntegraciГіn con la regla permanente Skills Inmutables:**

La regla permanente establece que solo StandardsAgent puede proponer cambios a `/skills/`, con gate SecurityAgent + confirmaciГіn humana. En v4.0, `register_skill()` solo puede ser invocado por un agente con permiso `skill:write` en `PermissionStore`, que solo se concede a StandardsAgent y expira en 30 minutos tras la confirmaciГіn humana.

## 12. ASYNCLOCKMANAGER вҖ” RACE CONDITIONS

```python
# sdk/piv_oac/core/async_lock.py

class AsyncLockManager:
    """
    Previene race conditions en ejecuciГіn paralela de agentes.
    Necesario para: merges de ramas, acceso a StateStore, escritura de logs.
    """

    def __init__(self):
        self._locks: dict[str, asyncio.Lock] = {}

    @asynccontextmanager
    async def acquire(self, resource: str):
        if resource not in self._locks:
            self._locks[resource] = asyncio.Lock()
        async with self._locks[resource]:
            yield
```

**Recursos protegidos por lock:**
- `gate:<task_id>:<gate_type>` вҖ” ejecuciГіn de gate (evita doble evaluaciГіn)
- `merge:<branch>` вҖ” operaciГіn de merge git
- `state:<objective_id>:<key>` вҖ” escritura en StateStore
- `log:veracidad` вҖ” escritura en logs_veracidad/

## 12b. CONTEXT SCOPE PROTOCOL (CSP)

### El problema real del gasto de tokens

El mayor gasto de tokens en PIV/OAC no estГЎ en lo que los agentes *dicen* entre sГӯ (sus outputs ya son estructurados y cortos gracias a los checklists de `contracts/gates.md`), sino en lo que *reciben* como input. En Gate 2b, tres agentes revisan el mismo diff en paralelo. Sin CSP, cada uno recibe el diff completo aunque su checklist solo cubra una fracciГіn de Г©l.

El CSP no inventa un nuevo idioma. Aplica lazy loading al input de los agentes de gate.

### Principio de diseГұo

**Un agente de gate solo recibe los fragmentos del artefacto relevantes para su checklist especГӯfico.** El artefacto completo se almacena una vez en `StateStore` con un `artifact_ref`. Cada agente recibe el `artifact_ref` + su secciГіn filtrada. Si necesita mГЎs contexto, lo solicita explГӯcitamente por `artifact_ref`.

**El razonamiento es in-agent.** El Chain of Thought de un agente no se transmite al siguiente. Solo viaja el veredicto estructurado + lista de hallazgos (ya definidos en `contracts/gates.md`). Esto ya estaba implГӯcito en los checklists вҖ” el CSP lo hace explГӯcito como regla de SDK.

### Filtros por agente en Gate 2b

```
Artefacto: diff completo de feature/<tarea>
Almacenado una vez en StateStore вҶ’ artifact_ref: sha256_del_diff

SecurityAgent recibe:
  artifact_ref + scope_filter: ["auth", "crypto", "input_validation",
                                "secrets", "session", "permissions"]
  вҶ’ solo archivos/lГӯneas que matchean el scope (aprox. 25-35% del diff)

AuditAgent recibe:
  artifact_ref + scope_filter: ["business_logic", "tests", "rf_coverage"]
  вҶ’ archivos de lГіgica de negocio + tests (aprox. 35-45% del diff)

StandardsAgent recibe:
  artifact_ref + scope_filter: ["tests", "docstrings", "imports", "linting"]
  вҶ’ archivos de tests + mГіdulos pГәblicos (aprox. 25-40% del diff)
```

Si un agente detecta que necesita secciones fuera de su scope (e.g., SecurityAgent encuentra una anomalГӯa en lГіgica de negocio), solicita el fragmento adicional por `artifact_ref`. El SDK lo inyecta en el siguiente turno. Sin `artifact_ref` no hay acceso al artefacto вҖ” no se pueden pasar diffs completos en el prompt.

### ImplementaciГіn en el SDK

```python
# sdk/piv_oac/core/csp.py

# Scopes canГіnicos por agente de gate вҖ” derivan de contracts/gates.md
AGENT_SCOPES: dict[str, list[str]] = {
    "SecurityAgent": [
        "auth", "authentication", "authorization", "crypto", "bcrypt",
        "jwt", "token", "secret", "password", "session", "rbac",
        "input_validation", "sanitize", "permission", "cors", "header",
    ],
    "AuditAgent": [
        "business_logic", "domain", "service", "repository",
        "tests", "test_", "spec_", "rf_coverage",
    ],
    "StandardsAgent": [
        "test_", "_test", "spec_", "docstring", "import",
        "public_api", "README", "docs/",
    ],
    "CoherenceAgent": [
        # CoherenceAgent opera sobre diffs de subramas, no sobre el diff de tarea.
        # Recibe directamente los diffs inter-experto vГӯa su protocolo propio (contracts/gates.md В§Gate 1)
    ],
}

@dataclass
class ScopedArtifact:
    artifact_ref: str           # SHA-256 del artefacto completo en StateStore
    filtered_content: str       # Solo las lГӯneas relevantes para este agente
    scope_applied: list[str]    # QuГ© filtros se usaron
    filtered_pct: float         # Porcentaje del artefacto que se enviГі (para mГ©tricas)
    full_artifact_available: bool = True  # El agente puede pedir mГЎs si lo necesita

class ContextScopeProtocol:
    """
    Filtra el artefacto a revisar segГәn el scope del agente receptor.
    Se ejecuta en GateEnforcer antes de inyectar el artefacto en el prompt del agente.
    """

    def __init__(self, state_store: StateStore):
        self._store = state_store

    async def prepare_for_agent(
        self,
        artifact_ref: str,
        agent_type: str,
    ) -> ScopedArtifact:
        """
        Carga el artefacto desde StateStore y filtra al scope del agente.
        """
        # Cargar artefacto una sola vez desde StateStore (compartido entre agentes paralelos)
        full_content = await self._store.load(f"artifact:{artifact_ref}")
        if not full_content:
            raise ArtifactNotFound(f"artifact_ref '{artifact_ref}' no encontrado en StateStore")

        scope = AGENT_SCOPES.get(agent_type, [])
        if not scope:
            # Agente sin scope definido recibe el artefacto completo
            return ScopedArtifact(
                artifact_ref=artifact_ref,
                filtered_content=full_content,
                scope_applied=[],
                filtered_pct=1.0,
            )

        filtered = self._filter_by_scope(full_content, scope)
        filtered_pct = len(filtered) / max(len(full_content), 1)

        return ScopedArtifact(
            artifact_ref=artifact_ref,
            filtered_content=filtered,
            scope_applied=scope,
            filtered_pct=filtered_pct,
        )

    def _filter_by_scope(self, content: str, scope: list[str]) -> str:
        """
        Filtra lГӯneas del diff que son relevantes para el scope.
        Retiene: lГӯneas con match de scope + N lГӯneas de contexto alrededor.
        """
        lines = content.split("\n")
        relevant_indices: set[int] = set()
        context_lines = 5  # lГӯneas de contexto alrededor de cada match

        for i, line in enumerate(lines):
            if any(keyword.lower() in line.lower() for keyword in scope):
                # Incluir contexto alrededor del match
                for j in range(max(0, i - context_lines), min(len(lines), i + context_lines + 1)):
                    relevant_indices.add(j)

        if not relevant_indices:
            # Sin matches en el scope вҶ’ retornar artefacto completo con advertencia
            return f"[CSP: sin matches para scope {scope} вҖ” artefacto completo]\n{content}"

        filtered_lines = [lines[i] for i in sorted(relevant_indices)]
        return "\n".join(filtered_lines)

    async def store_artifact(self, content: str) -> str:
        """
        Almacena el artefacto en StateStore y retorna su artifact_ref.
        Se llama una vez por GateEnforcer antes de lanzar los agentes paralelos.
        """
        artifact_ref = hashlib.sha256(content.encode()).hexdigest()
        await self._store.save(
            f"artifact:{artifact_ref}",
            content,
            ttl=timedelta(hours=2),  # Los artefactos de gate expiran en 2 horas
        )
        return artifact_ref
```

### IntegraciГіn con GateEnforcer

```python
# ModificaciГіn a gate_enforcer.py вҖ” Gate 2b con CSP

async def enforce_gate_2b(self, task_id: str, diff_content: str, agents: list):
    # 1. Almacenar artefacto una sola vez (no repetir en cada agente)
    artifact_ref = await self._csp.store_artifact(diff_content)

    # 2. Preparar scope por agente вҖ” en paralelo
    scoped = await asyncio.gather(*[
        self._csp.prepare_for_agent(artifact_ref, agent.agent_type)
        for agent in agents
    ])

    # 3. Lanzar agentes con su scope filtrado
    verdicts = await asyncio.gather(*[
        agent.review_with_scope(scoped_artifact)
        for agent, scoped_artifact in zip(agents, scoped)
    ])

    # 4. Registrar mГ©tricas de filtrado en ExecutionAuditor
    for agent, sa in zip(agents, scoped):
        self._execution_auditor.record_event(ExecutionEvent(
            type=EventType.CSP_FILTER_APPLIED,
            agent_id=agent.id,
            filtered_pct=sa.filtered_pct,
            scope=sa.scope_applied,
        ))

    return self._consolidate(verdicts)
```

| Regla | DescripciГіn |
|---|---|
| **Sin Bypass de Gate** | `gate:*:bypass` no es un permiso vГЎlido en PermissionStore. NingГәn grupo ni usuario puede concederlo. Los gates solo pueden aprobarse por agentes responsables o confirmaciГіn humana (Gate 3). |

### Riesgos residuales y mitigaciones

| Riesgo | MitigaciГіn |
|---|---|
